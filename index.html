This project proposes a note-centric browser that manages tabs, notes, PDFs, and pages like a cognitive system. Instead of keeping everything equally active, it learns what should stay visible, what should fade, and what should move out of the working set.

Motivation

Why we come up with this idea

Today, most knowledge work happens in digital environments where we have too many tabs, notes, and documents open at the same time. However, human working memory is limited. When too many things compete for attention, people experience cognitive overload, which leads to more task switching, slower progress, and more mistakes. Most existing tools try to solve this problem by helping users store and remember more information. This project explores the opposite idea: selective forgetting may be just as important for cognitive efficiency as remembering.

Too Many Active Items

Tabs, notes, webpages, and PDFs all compete for limited mental resources.

Working Memory Is Limited

Remembering Is Not Enough

How the System Will Be Built?

This interface consists of a side-panel workspace that organizes all items, including tabs, pages, and notes, into two layers: working set and Library.

🗂️

Working Set

keep current tasks, represents actively used information (working memory)

📂

Library

stores archived information (long-term memory)

🏷️

Item Blueprint

Each item is represented by a structured blueprint containing attributes(id, type, state, priority, strength, tag_list, last_access_time, action_history_list).

👤

User Control

Users can pin, restore, override suggestions, or turn adaptive mode on and off.

The methods we applied in this system

The system combines a base decay model with reinforcement learning so it can both estimate relevance and improve its suggestions through feedback.

01

Base Decay Model

Applied Ebbinghaus Forgetting Curve. Every item has a strength value that decays over time when it is not used(Figure 1). Clicking, editing, searching, and scrolling reinforce that item and raise its activation again (Figure 2).

02

Recommendation

Based on strength and priority, the system suggests one of four actions: highlight, deprioritize, archive, or prune. Suggestions appear beside tabs as actionable controls rather than disruptive pop-ups.

03

Reinforcement Learning

The RL agent compares its suggestions with what the user actually does. Accepted suggestions become positive feedback; rejected suggestions or later restores become negative feedback.

04

Update Cycles

The system employs two update mechanisms

Time-based update: Runs at fixed intervals during use and performs a global update at night. This process decays the strength of each item over time and generates updated workspace recommendations.

Post-session update: Runs after each working session. The system records user actions during the session and updates item states and RL policy based on user feedback.

Evaluation

How success is tested

The system is evaluated by asking whether it reduces the working memory burden of active-item management while preserving task performance.

Condition Description Why It Matters
Standard Browser No adaptive support. Provides a baseline for cognitive overload.
Static Organization Tool Manual organization without AI. Tests whether improvement comes from organization alone.
Full System Adaptive forgetting with machine learning. Tests the full contribution of decay plus RL.

Impacts

benefits & downsides

This project contributes to the psychology and computer science streams of the cognitive systems major by leveraging adaptive forgetting from psychology into an algorithmic assistant to optimize the cognitive load of the user.

Positive Impact

For research-heavy work, the system could reduce tab piles, lower mental effort, cut down context switching, and make long digital projects feel more manageable. At a broader level, it points toward software that respects attention limits.

Risks and Concerns

Fine-grained interaction histories can reveal habits and interests, so privacy is a serious concern. Wrong recommendations can also interrupt flow or feel unpredictable if users do not feel in control.

The system should feel like an assistant, not a manager. Users need to be able to disable learning, limit pruning, restore content, and decide how much initiative the program can take.

  • Disable learning entirely
  • Archive-only mode without pruning
  • Automation caps for specific contexts
  • Clear restore paths and transparent reasons for actions

Future versions could explain why a tab was archived or deprioritized, and could potentially expand beyond the browser into broader operating-system support, though that would also increase privacy concerns.
